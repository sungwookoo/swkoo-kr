# Per-user registrations (Phase 3.1+)

Each `<github-login>.yaml` here is a *registration pointer* generated by
`swkoo.kr/deploy` when a friend hits Deploy. The file is intentionally
small — it just names the per-user *deploy* repo (in the `swkoo-deploy`
GitHub org) where the actual k8s manifests live.

Schema (current as of 2026-05-21):

```yaml
login: <github-login>
appName: <repo-name-sanitized>
deployRepo: swkoo-deploy/<github-login>   # k8s manifests live here
sourceRepo: <github-login>/<repo>         # Dockerfile + GHA workflow live here
image:
  repo: ghcr.io/<github-login>/<repo>
  updateStrategy: digest
  scanResult: <pending|...>
  writeBackMethod: argocd-image-updater
```

`sourceRepo` was added 2026-05-21 so Observatory's pipelines.service can
find the right GitHub Actions runs (the deploy repo has only manifests,
no workflows).

The `swkoo-users` ApplicationSet (`deploy/argocd/users-applicationset.yaml`)
reads this directory via the `files: deploy/users/*.yaml` generator and
materializes one Argo CD Application per friend (`swkoo-user-<login>`).
The Application's `swkoo.kr/source-repo` annotation carries `sourceRepo`
through to runtime.

## Friend onboarding

Self-serve through `https://swkoo.kr/deploy` — operator no longer
manually copies templates. The backend (`apps/backend/src/deploy/`)
generates these registration files plus the manifests in the
`swkoo-deploy/<login>` repo on commit.

## Do not edit manually

Hand edits get overwritten the next time the user hits Deploy. Use the
backend register flow (or, for emergency fixes, edit the deploy repo
directly at `swkoo-deploy/<login>`).
